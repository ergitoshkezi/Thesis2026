1. Project Overview

This thesis project is a specialized software tool designed to parse, visualize, and analyze Siemens enterprise system transaction logs. These logs are produced by a Siemens proprietary runtime environment and record the execution of business transactions involving:

  • CDOs (Configurable Data Objects) — reusable business object definitions
  • CLFs (Custom Logic Functions) — custom code modules implementing business rules
  • XML Request/Response documents — the input/output data of each transaction

The problem being solved is that raw Siemens log files are very large (often hundreds of megabytes), difficult to read due to timestamp encoding and nested depth levels, impossible to compare visually when debugging transaction differences, and time-consuming to analyze manually for performance bottlenecks. The tool provides automated parsing, visual tree representation, AI-assisted analysis, cross-file comparison, and Excel-based reporting — all within a desktop GUI application.

4. Log File Format Explained

4.1 Transaction Section

YYYYMMDDHHMMSS.mmm  <ThreadID>  Depth: <N>  CDO: <Name>  <Operation>: <Value>

Example:

20240315143022.123  0x12A4  Depth: 2  CDO: OrderService  Perform: CreateOrder  Exec: validateInput  param1 = "ABC",  resolved value = ( "ABC" )
FieldDescription
YYYYMMDDHHMMSS.mmmTimestamp — 14 digits + 3 millisecond digits
ThreadIDHex identifier of the execution thread
Depth: NNesting depth of the CDO/CLF call stack
CDO: NameThe Configurable Data Object being called
OperationType of action (Perform, Field, Exec, etc.)
Leaving Depth: NMarks the end of a nested operation at depth N

4.2 Request Document Section

Begins with "Request Document" and ends at "starting transaction". Contains the XML payload sent to trigger the transaction. The parser skips this section for the tree view but can extract and save it to a separate XML file.

4.3 Response Document Section

Begins with "Response Document" and contains the XML result. The parser can extract this for XML timing analysis.

4.4 State Machine for Parsing

state = {
    'in_request_document': False,
    'in_transaction': False
}

This ensures lines inside a transaction body are parsed and added to the tree, XML sections are separated and saved to their own files, and non-transaction lines are silently skipped.

5.3 Function Reference

main() — Entry Point

The main batch processing loop: opens a folder selector dialog, counts .txt/.log files to configure the progress bar, then for each file resets all globals, reads all lines with read_lines_list(), iterates line-by-line calling Analyse(), and commits results to Excel with append_to_excel().

read_lines_list(filename) — Adaptive File Reader

Reads raw file content using one of two strategies: simple binary read + chardet decode for files under 1 MB, or mmap (memory-mapped file) for zero-copy reading on files 1 MB and above. Returns a list of non-empty stripped strings.

Analyse(token_of_row, row_index_log) — Core State Machine

Analyse()
 ├── "<" + "?" → xml_Request_funzionante()  [start of XML Request]
 ├── "starting" + "transaction" → start() + thread()
 ├── "Depth" (not "Leaving") → Total_CDO += 1
 ├── "Response" + "Document" → mark response section start
 ├── "Request" + "Document" → mark request section start
 ├── "deleting" + "transaction" → end() + delta_txn() + append gap tracking
 ├── "EXCEPTION" / "csiXMLException" → ERROR = True
 └── "Description" → Error_Description()

gap_time(token, index) — Biggest Gap Tracker

Uses a two-pointer sliding window to find the maximum time gap between any two consecutive timestamped log lines. Identifies the line in the transaction where the system was "stuck" the longest — a critical performance diagnostic.

append_to_excel() — Excel Writer

Writes one row to _Overview.xlsx using pandas with openpyxl engine. Appends to an existing file or creates a new one with correct column headers.

6. Module 2 — Log_Parser_Ultimate.py (Interactive GUI Viewer)

6.1 LLMApiRunnable Class

A minimal HTTP client for calling the Siemens LLM API at https://api.siemens.com/llm/v1/chat/completions. Posts a JSON payload with model "mistral-7b-instruct", extracts choices[0].message.content from the response, and is callable directly via __call__ delegation to invoke().

6.2 Global Utility Functions

split_text_no_overlap(text, max_tokens=7500)

Splits a text string into chunks of at most 7500 NLTK tokens, working from end to beginning, then reversing the result. This gives priority to the most recent (most relevant) log entries. Used by SiemensGPT().

find_matching_sublists(file_, e, diff_rows)

An O(n + m) string matching algorithm using a dictionary. Marks each extra row as [FOUND] (content exists in the other file at a different position) or [NOT FOUND] (genuinely new content).

tokenize(file1, file2, row, diff_rows) — comparison version

Token-level diff for two log lines. Skips the first 3 tokens (timestamp, threadID, second token), removes structural noise (=, :), wraps changed tokens with @@ for highlighting, and sets flag=True if all tokens differ.

parse_log(log_file, file) — Main Parsing Engine ⭐

The most critical method in the application. Reads all log lines via read_lines_list_XML(), starts async XML processing in a background thread, then processes each line detecting three patterns via pre-compiled regex:

Pattern A: TIMESTAMP_PATTERN
  → Extracts timestamp, computes delta since last timestamp
  → Maintains top-5 largest deltas in self.refference[]

Pattern B: DEPTH_PATTERN ("Depth: N")
  → Enters a new tree node as a CHILD of the current stack top
  → Pushes node_id onto the deque stack; tags "green_text"

Pattern C: LEAVING_PATTERN ("Leaving Depth")
  → Creates a "Leaving" node as sibling of the parent
  → Pops from the stack, calculates CDO execution time
  → Updates the parent node's "CDO DeltaTime" column; tags "orange_text"

UI updates are batched in groups of 1000 operations (_process_batch()) to avoid per-line GUI redraws.

8. AI Integration — SiemensGPT

8.1 Architecture

The AI layer uses the Siemens internal LLM API serving the mistral-7b-instruct model via standard REST (POST with JSON body).

8.2 SiemensGPT(log, question, api_key)

Chunks the log into ≤7500-token pieces (working backward from the end to prioritize recent entries), sends each chunk with a detailed prompt explaining the log format and the user's question, then uses Resume_GPT() to aggregate responses when ≥3 chunks are processed.

8.3 compareGPT(log, api_key)

Called automatically during file comparison. Sends [FOUND]/[NOT FOUND] annotated extra rows to the AI, which explains the semantic meaning of the differences.

8.4 Resume_GPT(log, api_key)

A meta-call that takes concatenated results of multiple chunk analyses and produces a single coherent summary — a "reducer" in the map-reduce pattern.

8.5 Prompt Engineering

Every prompt includes domain knowledge context: CDO types (Service CDO vs. Data CDO), CLF explanations, variable expression formats, resolved vs. unresolved values, and instruction to treat "deleting transaction" as a terminal marker only.

9. File Comparison Engine

9.1 Algorithm

A synchronized linear scan (not a traditional LCS/diff algorithm):

row = 0
while row < max(len(file1), len(file2)):
    if row >= len(file1): collect remaining file2 rows as e2; break
    if row >= len(file2): collect remaining file1 rows as e1; break
    r2, diff_rows, flag = tokenize(file1[row], file2[row], row, diff_rows)
    if flag: insert " - " placeholder into file1 at current position
    diff_file1.append(file1[row])
    diff_file2.append(timestamp_prefix + r2)
    row += 1

9.2 Token-Level Diffing

Changed tokens are wrapped with @@ markers, displayed as green highlights. This pinpoints exactly which field changed (e.g., a resolved value changed from "A" to "B").

9.3 Extra Row Classification

Extra rows go through find_matching_sublists(): content is looked up in a dict built from the other file's rows. If found → [FOUND] (action exists but at a different time); if not found → [NOT FOUND] (completely new action).

10. XML & JSON Analysis Pipeline

10.1 create_xml_file() — Async XML Builder

Runs in background thread: parses XML with ET.fromstring(), pretty-prints with minidom, saves as {filename}.xml, then calls process_node() recursively to build a JSON tree. Each node is annotated with node_type, level, rank (top-3 slowest at its level), delta_time_ms, percentage_of_level, and subnodes.

10.2 Analyze_XML() — Query Extractor

Searches the XML tree for all nodes whose tag contains "__query", finds the nearest ancestor with timing information, and adds them to all_query_data sorted by delta_time_ms (slowest first).

10.3 only_show_XML_TreeView() — Direct XML Mode

Used for raw .xml files. Parses with ET.parse(), inserts each node into the TreeView, tracks timing per level, and colors the top-3 slowest nodes in red at each level.

10.4 show_XML_TreeView() — JSON Mode

Reads a pre-computed _tree_analysis.json and populates the TreeView. Supports full-text search, "Expand Critical Nodes" (auto-expands all rank_1 nodes), and "Analyze Queries".

10.5 show_Result_popup() — Query Analysis Report

Renders the top-40 slowest queries in a scrollable canvas, with node type, timing, percentage, "Copy Query" button for SQL text, and recursive subnode display. Warns if more than 40 queries were found.

15. Error Handling Strategy

  • Exception detection in logs: Analyse() detects EXCEPTION, csiXMLException, and csiException tokens, sets ERROR = True, and extracts the description for the Excel Exception column.
  • Runtime GUI errors: Almost all methods wrap logic in try/except Exception as e: blocks and show messagebox.showerror() dialogs to prevent crashes.
  • Encoding errors: Both file readers handle UnicodeDecodeError silently to accommodate files with mixed or corrupt encoding segments.
  • Batch processing errors: _process_batch() catches tk.TclError for individual node operations, allowing the rest of the batch to proceed.
  • API errors: LLMApiRunnable.invoke() catches KeyError/IndexError and returns a descriptive error string instead of crashing.
  • XML parse errors: create_xml_file() catches ET.ParseError and reports it while allowing the rest of the application to continue.

18. Design Decisions & Technical Rationale

Why Two Separate Files?

Structure_Excel.py handles purely batch, non-interactive processing and can be run standalone from the command line. Log_Parser_Ultimate.py handles interactive analysis. Separation keeps concerns distinct and allows Structure_Excel.main() to be invoked as a library call from the GUI.

Why Tkinter + PyQt6 Mixed?

The core GUI is in tkinter (simpler, no external dependency). However, tkinter's built-in progress dialog is limited and not always modal. PyQt6's QProgressDialog is modal and platform-native, used specifically for long-running operations.

Why Global Variables in Structure_Excel?

The batch parser works in a single execution thread. Using globals avoids parameter-passing overhead across deeply nested calls. While not recommended for library code, it is pragmatic for a single-purpose script module.

Why the Deque/Stack Approach for Tree Building?

Siemens log files express call stack depth explicitly through Depth: N tokens. A deque as a stack perfectly mirrors the call stack. When Leaving Depth is detected, the CDO delta time is computed by popping the node and measuring elapsed time.

Why NLTK Tokenization for Comparison?

NLTK's word_tokenize correctly handles punctuation as separate tokens, allowing the comparison to separate structural characters from semantic values for accurate token-level diffing.

Why Chunked AI Processing?

The Mistral-7B model has a finite context window. Large log files can easily exceed 15,000+ tokens. Chunking to 7500 tokens stays within safe limits. Resume_GPT() then synthesizes multiple chunk analyses into one — implementing a map-reduce pattern for AI-assisted analysis.
